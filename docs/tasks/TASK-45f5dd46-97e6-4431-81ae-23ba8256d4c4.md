# Task: Audit Logging for Enterprise Customers

## Goal

**Tenant admins** (lugia) need a searchable, filterable log of all security-relevant activity within their organization — who did what, when, and from where. This is a core enterprise feature required for SOC 2 (CC6.1, CC7.2, CC8.1) and ISO 27001 (A.12.4) compliance.

**Internal admins** use the same lugia audit log screen (lugia and giratina share auth, with additional checks for giratina). No separate giratina API is needed — internal admins can view any tenant's audit log through lugia.

**Before:** No visibility into user actions. Tenant admins cannot investigate incidents or demonstrate compliance.

**After:** Tenant admins see a paginated, filterable audit log of all mutations, auth events, and access failures within their tenant. Exportable to CSV for SIEM integration.

## Background

### Compliance drivers

SOC 2 and ISO 27001 require logging of:

- Authentication events (login success/failure, logout, token refresh)
- Authorization failures (permission denied, feature gate blocked)
- All user-initiated mutations (CRUD on roles, users, IP whitelist, tenant settings)
- Security configuration changes (SSO, IP whitelist activation, enterprise feature toggles)

GDPR (Article 30) requires logging access to personal data:

- Who accessed personal data, when, and from where
- Modifications to personal data (covered by mutation logging above)
- In our app, personal data (email, name) is only visible on the users list page and in the audit log viewer itself. We log access to the users list; we do not log access to the audit log viewer (recursive and impractical).

Not required in audit logs (these belong in system/ops logs):

- Server errors (500s, DB timeouts) — system logs
- Validation errors (400s, bad input) — not security-relevant

### Audit logs vs system logs (jirachi/logger)

These are separate systems for different audiences:

|                            | System logs (jirachi/logger) | Audit logs (new)           |
| -------------------------- | ---------------------------- | -------------------------- |
| **Audience**               | Our ops team                 | Tenant admins / compliance |
| **Storage**                | stdout → Cloud Logging       | PostgreSQL                 |
| **Queryable by customers** | No                           | Yes                        |
| **Includes system errors** | Yes                          | No                         |

Both may log the same event (e.g., login) but for different consumers. Keep both.

### Action naming — distinct from permissions

The RBAC permission system uses `resource` + `action` where actions are coarse (`view`, `edit`). Audit log actions are more granular. To avoid confusion:

| System      | Resource style    | Action style                           | Example            |
| ----------- | ----------------- | -------------------------------------- | ------------------ |
| Permissions | plural (`roles`)  | capability (`view`, `edit`)            | `roles` + `edit`   |
| Audit logs  | singular (`role`) | past-tense verb (`created`, `deleted`) | `role` + `created` |

The audit log table has two separate columns: `resource_type` (e.g., `role`, `user`, `auth`, `access`) and `action` (e.g., `created`, `invited`, `login`). The dotted notation in the events list below (e.g., `role.created`) is shorthand for `resource_type=role, action=created`.

New permission for viewing audit logs: `audit_log` + `view` (follows existing permission pattern).

### Events to log

**Auth events** (only logged when user exists — see "Pre-auth events" in Architecture):

- `auth.login` (outcome: success/failure)
- `auth.logout`
- `auth.password_changed`, `auth.password_reset_requested`, `auth.password_reset_completed`

**Access events** (always outcome: failure):

- `access.permission_denied`
- `access.feature_gate_blocked`
- `access.ip_blocked`

**User management:**

- `user.invited`, `user.deleted`, `user.email_changed`
- `user.roles_updated`, `user.invite_resent`

**GDPR read access:**

- `user.list_viewed` — logged when the users list page is accessed (the only page displaying personal data). Not logged for the audit log viewer itself (would be recursive).

**Role management:**

- `role.created`, `role.updated`, `role.deleted`

**IP whitelist:**

- `ip_whitelist.activated`, `ip_whitelist.deactivated`
- `ip_whitelist.ip_added`, `ip_whitelist.ip_removed`, `ip_whitelist.ip_updated`
- `ip_whitelist.emergency_deactivated`

**Tenant management:**

- `tenant.name_changed`
- `tenant.enterprise_feature_toggled`

## Architecture

### Data flow

```
Handler (within same DB transaction as the mutation):
  → performs mutation
  → calls audit log INSERT as part of the same transaction
  → transaction commits atomically (both mutation + audit log, or neither)

Middleware/read handlers (access failures, GDPR read events — no mutation transaction):
  → calls audit log INSERT directly (standalone write, acceptable to lose)

Read path:
  → GET /audit-logs → SQLC query filtered by tenant_id + user filters → paginated response
  → Frontend renders table with filters, CSV export
```

### Key design decisions

**Transactional writes:** Audit log inserts for mutations MUST be part of the same database transaction as the mutation. If the mutation succeeds but audit log fails (or vice versa), the system is inconsistent. The writer function takes a `pgx.Tx`, not a standalone pool. For middleware-level events (access failures), there's no mutation transaction — a standalone insert is acceptable since losing an access failure log is a minor gap, not data inconsistency.

**Append-only:** No UPDATE or DELETE queries on `audit_logs`. Immutability enforced at the application layer.

**actor_id is NOT NULL:** Our app soft-deletes users (anonymizes email/name, sets `deleted_at`). The user row persists, so the FK never breaks. Store `actor_name` and `actor_email` in metadata at write time so the audit log remains readable even after the actor's user row is anonymized.

**Pre-auth events:** Login failures for non-existent emails cannot be logged to the audit log — there's no tenant_id, no actor_id, and no enterprise feature flag to check. These are logged to system logs only (jirachi/logger). Login failures for existing users (wrong password) ARE logged — we have the user and can look up their tenant. For auth events with known users, the handler looks up the user by email (which gives tenant_id), then checks the enterprise feature flag before writing the audit log.

**Cross-tenant actor (internal admin actions):** When an internal admin performs an action on tenant X (e.g., toggling enterprise features), the audit log records `tenant_id = X` with the admin's `actor_id`. The users table is global, so the FK is valid even though the admin belongs to a different tenant. Metadata should include `{"is_internal_admin": true}` to make this visible.

**resource_type and action are separate columns:** `resource_type` (e.g., `role`, `user`, `auth`) enables filtering by category. `action` (e.g., `created`, `login`, `invited`) enables filtering by specific operation. Together they form the full event identifier.

**resource_id:** The UUID of the target of the action. Examples: the new role's UUID for `role.created`, the invited user's UUID for `user.invited`, NULL for system events like `auth.login`.

**metadata (JSONB):** Action-specific context. Examples: `{"role_name": "Editor"}` for `role.created`, `{"email": "new@example.com"}` for `user.invited`, `{"reason": "invalid_password"}` for `auth.login` with outcome `failure`. For v1, we filter by fixed columns only (actor_id, action, resource_type, outcome, date range) — not by metadata contents. "Filter by actor" means selecting from a list of user IDs, not free-text name search. Actor name/email in metadata is for display only. If a metadata field becomes a frequent filter, promote it to a column later.

**Separate `outcome` column:** `outcome VARCHAR(50) NOT NULL DEFAULT 'success'` — values: `'success'` or `'failure'`. Enables clean filtering by outcome across all event types (`WHERE outcome = 'failure'`) and by action regardless of outcome (`WHERE action = 'login'`). Access events (permission checks, feature gates, IP checks) are only logged on failure — successful access checks are high volume, low signal.

**Offset-based pagination:** Use the existing `pagination.PaginationMetadata` pattern (Page, Limit, Total, TotalPages, HasNext, HasPrev) — not cursor-based. Consistent with the rest of the codebase. For an append-only log ordered by `created_at DESC`, offset pagination is fine — new entries only appear at the top.

**IP address capture:** Use `lugia-backend/lib/iputils/iputils.go` (`ExtractClientIP`), which reads X-Forwarded-For rightmost for GCP Load Balancer. Do not use `r.RemoteAddr` directly.

**Enterprise gating:** Add an `AuditLog` struct with `Enabled bool` to `EnterpriseFeatures`, following the same pattern as `RBAC`, `IPWhitelist`, and `SSO`. Lugia gates both reads and writes behind this flag.

**SQLC query split:** Event types and constants live in jirachi (shared vocabulary). The INSERT query and a thin writer function live in each backend separately — this follows existing handler patterns where `qtx.InsertAuditLog(ctx, params)` works exactly like any other query on the same transaction. The SELECT/list query lives in lugia only (only lugia serves the read API). This means the INSERT SQL is duplicated across backends, but the duplication is minimal (one INSERT statement) and the benefit is that agents instrumenting handlers don't need to learn a new cross-module pattern.

**Permission seeding:** The `audit_log.view` permission row goes in the schema (fixed data, like existing permissions). For default role assignments and seed data, follow the existing pattern for other permissions. No production data migration needed — app is not live.

**/me endpoint integration:** The `/me` endpoint returns `ClientEnterpriseFeatures` (a subset of `EnterpriseFeatures` — currently only RBAC and IPWhitelist). The new `AuditLog` field must be added to `ClientEnterpriseFeatures` as well, not just `EnterpriseFeatures`. The `audit_log.view` permission already flows through the permissions array.

**Feature doc reference:** Every handler file and frontend page file related to audit logging must include the feature doc reference on line 1, following the existing codebase convention:

- Go files: `// Feature doc: docs/features/audit-logging.md`
- Svelte files: `<!-- Feature doc: docs/features/audit-logging.md -->`
- Files that instrument audit logging into existing handlers should add `docs/features/audit-logging.md` to their existing feature doc comment (comma-separated if they already reference another feature doc).

### What does NOT change

- `jirachi/logger/` — system logs remain separate
- Permission middleware — permission checks stay as-is
- Existing handler signatures — audit logging is additive
- RBAC permission model — we add one new permission row, don't change the model

## Security Considerations

- **Tenant isolation:** All queries MUST filter by `tenant_id` from JWT context. A tenant must never see another tenant's audit logs.
- **Immutability:** No UPDATE/DELETE queries. No API endpoint to modify or delete entries.
- **Deleted users:** Soft delete preserves user rows. Actor attribution in metadata survives anonymization. Audit trail is never lost — this satisfies both SOC 2 immutability requirements and GDPR (user data is anonymized in the users table, but the audit event record itself persists with the name/email as it was at the time of the action).
- **Permission gating:** Viewing audit logs requires `audit_log.view` permission (via existing RBAC).
- **No sensitive data in metadata:** Never store passwords, tokens, or PII beyond actor name/email for attribution.
- **CSV injection:** When exporting to CSV, sanitize metadata values that start with `=`, `+`, `-`, `@` to prevent spreadsheet formula injection. Prefix with a single quote or tab.

## Performance Considerations

- **Write path:** Synchronous within the mutation transaction. One extra INSERT per mutation — negligible latency for the atomicity guarantee.
- **Read path:** Primary query is `WHERE tenant_id = $1 ORDER BY created_at DESC` with offset pagination. Composite index covers this. Add indexes for all filter columns (tenant_id + created_at, tenant_id + actor_id, tenant_id + action, tenant_id + resource_type) to support filtered queries without sequential scans.
- **Table growth:** Grows indefinitely for v1. Future: partition by month, retention policies, cold storage archival.

## Discoveries

- **Middleware signature change**: `RequireFeature`/`RequireRBAC`/`RequireIPWhitelist`/`RequireAuditLog` needed a `db *queries.Queries` parameter added to support audit logging of access failures. All callers in `main.go` updated.
- **Giratina cross-module dependency**: Giratina cannot import `lugia/lib/iputils`. For the `update_tenant` handler, IP extraction uses `net.SplitHostPort(r.RemoteAddr)` directly instead.
- **Enterprise feature toggle audit logging**: When toggling enterprise features (including audit_log itself), we check the NEW feature state to decide whether to log. If audit_log is being enabled for the first time, that enabling event is logged. If disabled, the disabling event is not logged (acceptable trade-off).
- **Unauthenticated endpoints**: Login, forgot_password, and reset_password handlers need to load tenant enterprise features manually (by unmarshaling from the tenant record) since `LoadTenantAndUserContext` middleware isn't in their chain.
- **Test seed data**: Updated `seed.go` to include `audit_log: {enabled: true}` for enterprise tenant and `audit_log_view` permission to match `seed.sql`.
- **Giratina audit logging only covers customer compliance, not internal compliance**: Giratina's `update_tenant` handler logs to the _customer's_ audit_logs table, gated by the customer's `audit_log.enabled` feature flag. This means: (1) if the customer doesn't have audit logging enabled, internal admin actions on their tenant are not logged anywhere, and (2) there is no separate "internal admin audit log" for our own SOC 2 compliance. A follow-up task should address internal admin compliance logging — either by always writing to the customer's audit log regardless of feature flag, by creating a separate internal audit log, or by designating a system tenant for internal logs.
- **Tenant impersonation not logged to audit_logs**: Giratina's `LogInToTenant` handler (`log_in_to_tenant.go`) only logs to the structured logger (`logger.LogAuthEvent`), not to the customer's `audit_logs` table. When an internal admin impersonates a customer tenant, the customer has no visibility into this access. This is a compliance gap — tenant impersonation should appear in the customer's audit log so they can see when and from where an internal admin accessed their account.
- **CSV export only downloads current page**: The frontend CSV export generates a file from the rows currently visible on screen, not all matching rows. With pagination (e.g., 200 events at 50 per page), only 50 are exported. A dedicated backend `/audit-logs/export` endpoint that streams all matching rows as CSV would be the correct solution.
- **INNER JOIN in ListAuditLogs excludes deleted users**: The query uses INNER JOIN on users to get actor name. If a user is deleted, their audit log entries become invisible in the viewer (the data still exists in the table, and actor name/email are preserved in the metadata JSON column). Consider changing to LEFT JOIN with COALESCE from metadata if audit completeness for deleted users is required.
